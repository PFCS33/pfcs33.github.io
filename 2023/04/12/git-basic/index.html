After read this chapter, you can do all the basic local Git operations—creating or cloning a repo, making changes, staging and committing changes, viewing the history of all changes, working with remote and so on.

# Create a Git Repository

Tow way, end up with a Git repository on your local machine, ready for work

# From a local directory

  • You can take a local directory that is currently not under version control, and turn it into a Git repository
  • 流程
    • First , go to that project’s directory
    • Then type command git init
      • Will creates a new subdirectory named .git that contains all of your necessary repository files(Git repository skeleton)
  • E. do a initial commit
git add *.c 
git add LICENSE 
git commit -m 'Initial project version'

# clone

  • get a copy of an existing Git repository
    • Git receives a full copy of nearly all data that the server has.
      • By default, Every version of every file for the history of the project is pulled down.
  • COMMAND：git clone <url>
    • 除了HTTP，还可以使用其他transfer protocol，如SSH
  • E. git clone https://github.com/libgit2/libgit2
    • That creates a directory named libgit2, initializes a .git directory inside it, pulls down all the data for that repository, and checks out a working copy of the latest version
    • Rename directory: specify the new directory name as an additional argument
      • $ git clone https://github.com/libgit2/libgit2 mylibgit

# Ignoring Files

  • Deal with: files that you don’t want Git to automatically add or even show you as being untracked
  • create a file named： .gitignore
    • list the patterns to match the filename you want to ignore
    • RULES:
      • Standard glob patterns work
        • and will be applied recursively throughout the entire working tree.
      • # ：注释
      • You can start patterns with a / to avoid recursivity.
      • You can end patterns with a / to specify a directory.
      • You can negate a pattern by starting it with !
    • additional .gitignore files in subdirectories
      • apply only to the files under the directory where they are located

# Commtting Changes

  • COMMAND: git commit

    • Doing so launches your editor of choice, to edit your commit message
      • When you exit the editor, Git creates your commit with that commit message
          • the # ( comments and diff) stripped out
      • 自带注释信息：the latest output of the git status command
        • -v: puts the diff of your change
    • output
      • which branch you committed to + SHA-1 checksum
      • how many files were changed
      • statistics about lines added and removed in the commit

  • -m ：type your commit message inline

  • -a: skipping the staging area

    • Git will automatically stage every file that is already tracked before doing the commit

# Removing File

  • COMMAND: git rm
    • remove a file from Git = remove it from staging area + commit
      • also remove the file from your working directory
    • The next time you commit, the file will be gone and no longer tracked
    • You can pass files, directories, and file-glob patterns to the git rm command
    • NOTE: 如果你直接将文件从working directory删除，state会显示如下

  • -f
    • If you modified the file or had already added it to the staging area, you have to -f to force the removal
    • This is a safety feature to prevent accidental removal of data that hasn’t yet been recorded in a snapshot and that can’t be recovered from Git.
  • --cached
    • keep the file in your working tree, but remove it from your staging area(not have Git track it anymore)

# Moving File

  • Git doesn’t explicitly track file movement
    • Yet Git is pretty smart about figuring that out after the fact
  • COMMAND：git mv
    • can be used to rename a file
    • E.
      $ git mv README.md README 
$ git status 
On branch master 
Your branch is up-to-date with 'origin/master'. 
Changes to be committed:   
(use "git reset HEAD ..." to unstage)

      • As you can see, Git considers it a renamed file
      • Equivalent to:
        • you can use any tool you like to rename a file, and address the add/rm later, before you commit
      	$ mv README.md README 
	$ git rm README.md 
	$ git add README

# Undo Things

Be careful! because you can’t always undo some of these undos. This is one of the few areas in Git where you may lose some work if you do it wrong

  • COMMAND: git commit --amend
    • To amend your last commit
      • Actually, you replace the old entirely with a new & improved commit, not so much fix it
        • so it as if the previous commit never happend and it won't show up in your repository
      • It can make minor improvements to the last commit, without cluttering your repository history with commit messages like “Oops, forgot to add a file” or “Darn, fixing a typo in last commit"
      • E.
      $ git commit -m 'Initial commit' 
$ git add forgotten_file 
$ git commit --amend
    • ACTION: Just like the common commit comand, the same commit-message editor fires up, but it already contains the message of your previous commit.
      • You can edit the message the same as always, but it overwrites your previous commit
    • NOTE: PLEASE Only amend commits that are still local and have not been pushed somewhere
      • Amending previously pushed commits and force pushing the branch will cause problems for your collaborators
  • Unstaging a staged file
    • COMMAND: git reset HEAD <file>
      • make file unstaged & return to be modified
      • kind dangerous, but the file in your working directory is not touched, so it's relatively safe.
      • E.
      $ git add * 
$ git status On branch master

Remember, anything that is committed in Git can almost always be recovered. Even commits that were on branches that were deleted or commits that were overwritten with an --amend commit can be recovered

    • undo staging: git restore --staged

# Working with Remotes

  • Remote repository
    • a version of your project that is hosted on the Internet or network somewhere
      • also, can be on your local machine.
        • The word “remote” does not necessarily imply that the repository is somewhere else on the network or Internet, only that it is elsewhere
        • Working with such a remote repository would still involve all the standard pushing, pulling and fetching operations as with any other remote
      • You can have several of them, each of which generally is either read-only or read/write for you
  • To collaborating with others, you should know how to manage these remote repositories(add/remove repo, manage branch, define them as being tracked or not ...) & pushing and pulling data to share work

# Fetching & Pulling from Your Reomte

  • COMMAND: git fetch <remote>
    • Git will go out to that remote project and pulls down all the data from that remote project that you don’t have yet (since you cloned or last fetched from remote)
      • After you do this, you should have references to all the branches from that remote, which you can merge in or inspect at any time
    • It only downloads the data to your local repository — it doesn’t automatically merge
      • You have to merge it manually into your work when you’re ready.
    • E.
      • Paul’s master branch is now accessible locally as pb/master
      $ git fetch pb 
remote: Counting objects: 43, done. 
remote: Compressing objects: 100% (36/36), done. 
remote: Total 43 (delta 10), reused 31 (delta 5) 
Unpacking objects: 100% (43/43), done.

* [new branch] master -> pb/master  
* [new branch] ticgit -> pb/ticgit
  • COMMAND: git pull
    • automatically fetch and then merge that remote branch into your current branch

# Pushing to Your Remote

  • COMMAND: git push <remote> <branch>
    • push any commits you’ve done back up to the server
    • NOTE: This command works only if you cloned from a server to which you have write access and if nobody has pushed in the meantime
      • If you and someone else clone at the same time and they push upstream and then you push upstream, your push will rightly be rejected. You’ll have to fetch their work first and incorporate it into yours before you’ll be allowed to push

# Creating Tags

  • Git supports two types of tags: lightweight and annotated
    • lightweight: just a pointer to a specific commit
      • much like a branch that doesn’t change
    • Annotated: stored as full objects in the Git database
      • They’re checksummed; contain the tagger name, email, and date; have a tagging message; and can be signed and verified with GNU Privacy Guard (GPG)

# Deleting Tags

# Checking out Tags

  • You can do a git checkout of a tag
    • NOTE: this puts your repository in “detached HEAD” state
      • In this state, if you make changes and then create a commit, the tag will stay the same, but your new commit won’t belong to any branch and will be unreachable, except by the exact commit hash
      $ git checkout v2.0.0 
Note: switching to 'v2.0.0'. 

You are in 'detached HEAD' state. You can look around, make experimental changes and commit them, and you can discard any commits you make in this state without impacting any branches by performing another checkout. 

If you want to create a new branch to retain commits you create, you may do so (now or later) by using -c with the switch command. Example:   

git switch -c 

Or undo this operation with:   

git switch - 

Turn off this advice by setting config variable advice.detachedHead to false
