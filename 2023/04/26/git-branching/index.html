  • Git
  • Branching means you diverge from the main line of development and continue to do work without messing with that main line
  • Git branches is weight and fast
    • making branching operations nearly instantaneous
    • switching back and forth between branches generally just as fast

basically, branch in Git is a simple file which contains the 40 characters SHA-1 checksum of the commit it points to.(just need to add 40 characters and a new line)

# Branches in a Nutshell

  • Recall: How Git stores its data
    • a series of snapshots.
  • When you make a commit:
    • Git stores a object containing a pointer to the snapshot of your commit project (commit object)
    • this object also contains:
      • author’s name
      • email address
      • commit message
      • pointer to the parent commit (which directly came before this commit)
        • zero parents for the initial commit
        • one parent for a normal commit
        • multiple parents for a commit that results from a merge of two or more branches.
    • E. Assume that you have a directory containing three files, and you stage them and all commit.
      • Staging file, Git:
        1. computs the checksum for each one
        2. store that version of file in the Git repository(Git refers to them as blobs)
        3. adds that checksum to staging area
      • commit, Git
        1. checksums each subdirectory (in this case, just the root project directory)
        2. stores them as a tree object in the Git repository
        3. creates a commit object that has the metadata and a pointer to the root project tree
          • so it can re-create that snapshot when needed
      • Now, Git repository now contains five objects:
        • three blobs
          • each representing the contents of one of the three files
        • one tree
          • lists the contents of the directory and specifies which file names are stored as which blobs
        • one commit
          • with the pointer to that root tree
          • and all the commit matadata
      • if you make some changes and commit again, the next commit will store a pointer to the commit that came immediately before it.
  • A branch in Git is simply a lightweight movable pointer to one of these commits
    • The default branch name in Git is master.
      • As you start making commits, you’re given a master branch that points to the last commit you made
      • Every time you commit, the master branch pointer moves forward automatically.

      Note: The “master” branch in Git is not special. It is exactly like any other branch. The only reason nearly every repository has one is that the git init command creates it by default and most people don’t bother to change it

# Creating a New Branch

  • COMMAND: git branch <branch-name>
    • just creates a new pointer for you to move around.
      • It points to the same commit you're currently on.
    • It doesn't switch to the new branch.
    • E. $ git branch testing
  • How does Git know what branch you’re currently on?
    • It keep a special pointer called HEAD
      • a pointer to the local branch you’re currently on
      • In this case, you’re still on master branch
    • You can run git log command to shows you where the branch pointers are pointing. The option is called --decorate
      $ git log --oneline --decorate 
f30ab (HEAD -> master, testing) Add feature \#32 - ability to add new formats to the central interface 
34ac2 Fix bug \#1328 - stack overflow under certain conditions 
98ca9 Initial commit
      • You can see the master and testing branches that are right there next to the f30ab commit

# Switch Branches

  • NOTE: From Git version 2.23 onwards you can use git switch instead of git checkout to
    • git switch <branch-name>: switch to an existing branch
    • git switch -c <newbranch-name>: create & switch to it (-c stands for --create)
    • git switch -: return to your previously checked out branch
  • COMMAND: git checkout <branch-name>
    • It moves HEAD to point to the testing branch
    • COMMAND: git checkout -b <newbranch-name>
      • Creating a new branch and switch to it at the same time
  • What is the significance of switching branch?
    • you switch to a new working flow on testing branch, instead of the old master branch.
  • Let's do another commit
    $ vim test.rb
$ git commit -a -m 'made a change'
    • After this commit, testing branch will move forward, but your master branch stand still ,pointing to the previous branch
    • the command git commit will move forward the branch which HEAD points to.
      • So you diverge from the "master" workflow by switching branch.
  • Create a divergent project history
    • Then we switch back to master branch git checkout master
      • NOTE: In this command, Git did two things:
        • It moved the HEAD pointer back to point to the master branch
        • it reverted the files in your working directory back to the snapshot that master points to.
          • This also means the changes you make from this point forward will diverge from an older version of the project
          • It rewinds the work you have done in the testing branch, so you can go into different direction
      • NOTICE: Switching branches changes files in your working directory
        • If Git cannot do it cleanly, it will not let you switch at all
    • Let's make a few changes and commit again
      $ vim test.rb 
$ git commit -a -m 'made other changes'
      • Now your project history has diverged
        • As you can see, all changes are isolated in separate branches
        • you can switch back and forth between the branches, and merge them together when you’re ready
      • you can use git log to see the situation
      $ git log --oneline --decorate --graph --all
* c2b9e (HEAD, master) Made other changes 
| * 87ab2 (testing) Made a change 
|/  
* f30ab Add feature \#32 - ability to add new formats to the central interface 
* 34ac2 Fix bug \#1328 - stack overflow under certain conditions 
* 98ca9 initial commit of my project		```
  • COMMAND: git log
    • git log can show you the branch info, but by default, it only shows commit history below your current branch
    • To show commit history for the desired branch, you have to explicitly specify it.
      • git log <branch-name>
      • git log --all
  • NOTE: because the commit object are recording the parents info, finding a proper merge base for merging is automatically done for us and is generally very easy to do

# Basic Branching & Merging

  • Let's go through a simple & real world example of branching and merging
    1. Do some work on a website.
    2. Create a branch for a new user story you’re working on.
    3. Do some work in that branch.
    4. At this stage, you’ll receive a call that another issue is critical and you need a hotfix. You’ll do the following:
      1. Switch to your production branch.
      2. Create a branch to add the hotfix.
      3. After it’s tested, merge the hotfix branch, and push to production.
      4. Switch back to your original user story and continue working

# Basic Branching

  • First, assuming that you are working on your project and already made some commits on the master branch currently on.
  • Now, you decide to work on issue #53 (in whatever issue tracking system your company uses)
    • You need to create a new branch and switch to it
    • Use command git checkout -b or git switch -c
      $ git checkout -b iss53
Switched to a new branch "iss53"
  • Then you work on the website and do some commit, moving the iss53 forward
    $ vim index.html 
$ git commit -a -m 'Create new footer [issue 53]'
  • Now you are called to fix an issue hotfix immediately

    • With git, you can just switch to the mater branch, keeping changes you have made in issue53

      • Note that do switching need a clean work tree in your current working directory

    • Use git command git checkout

      • Now Git resets your work directory to the last time you commit on the master branch
      $ git checkout master 
Switched to branch 'master'

    • create a hotfix branch to this commit, and fix the bug in the hotfix branch

      $ git checkout -b hotfix 
Switched to a new branch 'hotfix' 
$ vim index.html 
$ git commit -a -m 'Fix broken email address' 
[hotfix 1fb7853] Fix broken email address  
1 file changed, 2 insertions(+)

  • Now you complete your task, and finally merge the hotfix branch back into master branch
    • COMMAND: git merge <branch-name>
      $ git checkout master 
$ git merge hotfix 
Updating f42c576..3a0874c 
Fast-forward 
index.html | 2 ++ 
1 file changed, 2 insertions(+)
    • Notice that there is "Fast-forward" in the prompt message
      • This means that Git just simply moves the master pointer forward to the commit which hotfix branch points to.
      • When you try to merge one commit into a commit which can be reached by the commit history, Git simplifies things by moving the pointer forward.
        • Direct ancestor
  • Now you can switch back to issue53 and continue your work

    • before you switch, delete the hotfix branch first, because you no longer need it.

      $ git branch -d hotfix 
Deleted branch hotfix (3a0874c).

    • and then:

      $ git checkout iss53 Switched to branch "iss53" 
$ vim index.html 
$ git commit -a -m 'Finish the new footer [issue 53]' 
[iss53 ad82d7a] Finish the new footer [issue 53] 
1 file changed, 1 insertion(+)

# Basic Merging

  • Now you decide that your issue53 work is complete and ready to be merge into your master branch
    • How: checkout the branch you wish to merge into, and then run git merge
      $ git checkout master 
Switched to branch 'master' 

$ git merge iss53 
Merge made by the 'recursive' strategy. 
index.html | 1 + 1 file changed, 1 insertion(+)
    • Notice it's different from hotfix example
      • because the branch you want to merge with (issue53) is not the direct ancestor of your branch (master)
        • In this case, your development history has diverged
    • Git will do a three-way merge, using the two snapshots the two branch point to and the common ancestor of two branch.
      • At the end, Git creates a new snapshot resulting from this three-way merge, and automatically creates a new commit pointing to it
      • This commit is referred to as a merge commit, special in having two parents
  • Then you can delete issue53 branch
    $ git branch -d iss53

# Basic Merge Conflicts

  • Occasionally, the process of issue53 example doesn't go smoothly
    • If you change the same part of the same file differently in the two branches you are merging, Git won't be able to merge them cleanly, and you will get a merge conflict.
      $ git merge iss53 
Auto-merging index.html 
CONFLICT (content): Merge conflict in index.html 
Automatic merge failed; fix conflicts and then commit the result.
    • git pauses the process of creating new commit and wait for you to resolve the conflict
  • Solve the conflicts
    • you can run git status to see which files are unmerged, at any point after the conflict.
    • Anything that has merge conflicts and hasn't been resolved is listed as unmerged
    $ git status 
On branch master You have unmerged paths.   
(fix conflicts and run "git commit") 

Unmerged paths:   
(use "git add ..." to mark resolution)   

	both modified: index.html 
	
no changes added to commit (use "git add" and/or "git commit -a")
    • And git adds standard conflict-resolution markers to files that have conflicts
      • you can open them manually and resolve those conflicts
      • E. your conflict file will contains a section like this:
      <<<<<<< HEAD:index.html
< div  id="footer">contact : email.support@github.com </div>
=======
< div  id="footer">
please contact us at support@github.com
</div>
>>>>>>>> iss53:index.html
      • This means that:
        • The version in HEAD is the top part of the block (above the =======)
        • The version in iss53 branch is the bottom part
      • In order to resolve the conflict, you need to either choose one side of part or merge it yourself
      • E. you can replace the entire block with this:
        • this has a little of each section,
        • and the <<<<<<< , ======= , >>>>>>> lines have been removed
        <div id="footer">
please contact us at email.support@github.com
</div>
  • After resolve conflicts in each conflict file, you can run git add on each conflict file to mark it as resolved in Git.
  • Note: you can also use a graphical tools to resolve these issues.
    • COMMAND: git mergetool

# Branch Management

  • COMMAND: git branch： more than create & delete
    • without argument: give a simple list of current branch
    • *: indicates the branch that you currently have checked out (HEAD points to)
      • also means that if you commit at this point, the master branch will be moved forward with your new work.
      $ git branch   
  iss53 
* master   
  testing
    • -v: see the last commit on each branch
      $ git branch -v   
  iss53 93b412c Fix javascript issue 
* master 7a98805 Merge branch 'iss53'   
  testing 782fd34 Add scott to the author list in the readme
    • --merged & --no-merged: see the branches that you have or have not yet merged into the branch you’re currently on
      • if you use --merged, Branches on this list without the * in front of them are generally fine to delete with git branch -d
      $ git branch --merged   
  iss53 
* master
      • if you use --no-merged, this shows your other branch
        • Because it contains work that isn’t merged in yet, trying to delete it with git branch -d will fail, you can force to do it with -D
        $ git branch -d testing 
error: The branch 'testing' is not fully merged. 
If you are sure you want to delete it, run 'git branch -D testing'.
      • you can add an additional argument <branch-name>, to ask about the merge state of this branch, avoiding checkout to this branch first
        $ git checkout testing 
$ git branch --no-merged master   
topicA   
featureB

# changing a branch name

Do not rename branches that are still in use by other collaborators. Do not rename a branch like master/main/mainline without having read the section "Changing the master branch name".

Note: this section tell about how to change branch name both locally and on the remote server

  • Locally:
    • COMMAND: git branch --move <old-name> <new-name>
      $ git branch --move bad-branch-name corrected-branch-name
  • Remote:
    • COMMAND: git push --set-upstream <remote-name> <new-name>
      $ git push --set-upstream origin corrected-branch-name
  • Let's take a brief look at all branches
    $ git branch --all 
* corrected-branch-name   
main   
remotes/origin/bad-branch-name   
remotes/origin/corrected-branch-name   
remotes/origin/main
    • Notice that the old branch name is still on the remote server, you can delete it with command:
      • git push <remote-name> --delete <old-branch-name>
      $ git push origin --delete bad-branch-name

# Change a master branch name

Changing the name of a branch like master/main/mainline/default will break the integrations, services, helper utilities and build/release scripts that your repository uses. Before you do this, make sure you consult with your collaborators. Also, make sure you do a thorough search through your repo and update any references to the old branch name in your code and scripts.

  • rename the master branch to main
$ git branch --move master main

$ git push --set-upstream origin main

$ git branch --all 
* main   
remotes/origin/HEAD -> origin/master   
remotes/origin/main   
remotes/origin/master
  • Now you have a few more tasks in front of you to complete the transition:
    • Any projects that depend on this one will need to update their code and/or configuration.
    • Update any test-runner configuration files.
    • Adjust build and release scripts.
    • Redirect settings on your repo host for things like the repo’s default branch, merge rules, and other thing s that match branch names.
    • Update references to the old branch in documentation.
    • Close or merge any pull requests that target the old branch.
  • After you’ve done all these tasks, you can delete the master branch
    $ git push origin --delete master

# Branching Workflows

In this section, we’ll cover some common workflows, so you can decide if you would like to incorporate them into your own development cycle.

# Long-Running Branches

  • Over a long period, merging from one branch into another will occur multiple times.
    • This means that you can have several branches that are always open and that you use for different stages of your development cycle, you can merge regularly from some of them into others
  • IDEA: having multiple long-running branches
    • These branches are at different levels of stability
      • when they reach a more stable level, they’re merged into the branch above them
  • For example: Most Git developers keep three workflows: three branches
    • master: code that is entirely stable
      • possibly only code that has been or will be released.
    • develop / next :
      • work form/ use to test stability
      • whenever it gets to a stable state, it can be merged into master
      • pull in topic branches when they’re ready
        • to make sure they pass all the tests and don’t introduce bugs
    • topic
      • some short-lived branches, like your earlier iss53 branch
  • isn’t necessary, but it’s often helpful, especially when you’re dealing with very large or complex projects.

# Topic Branches

  • a short-lived branch that you create and use for a single particular feature or related work
  • are useful in projects of any size
    • your work is separated into silos where all the changes in that branch have to do with that topic
    • You can did a few commits on them and deleted them directly after merging them into your main branch.
  • For Example:
    • you are doing some work (on master)

    • branching off the second branch to try another way of handling the same thing ( iss91v2)
    • going back to your master branch and working there for a while
